#### assunzioni su stato e parametri
possiamo individuare due tipi di assunzioni
- le invarianti di classe
- i contratti dei metodi

un invariante è una proprietà che lo stato di un'oggetto verificherà per tutta la sua esistenza
per poter scrivere i metodi di un oggetto attorno a questi invarianti
quando sviluppiamo un metodo e abbiamo un invariante di classe:
- possiamo assumere che sia rispettata la momento dell'invocazion
- garantire che l'invariante sarà rispettata alla terminazione dell'esecuzione

esempio: ogni istanza di `PuzzleBoard` è risolvibile
nel nostro esempio è facile verificarlo dato che nel nostro caso l'unico modo per alterare lo stato dell'oggetto è eseguire una mossa (valida)

#### i contratti
possiamo associare ad ogni metodo un contratto. Ci sono due parti, ogni una si impegna a raggiungere le aspettative contrattate

sui metodi questo rappresenta una garanzia sull'effetto derivante dall'esecuzione di un metodo quando questo viene invocato con parametri che soddisfano certe condizione
si parla di contratto perché:
- chi usa l'oggetto si impegna a passare i parametri corretti
- chi implementa il metodo si impegna a garantire i risultati attesi
possiamo trovare un esempio di contratto nei metodi della classe Object:
- `equals(Object o) : bool`
- `hashCode() : int`
- `toString() : String`
questi metodi sono implementati garantendo certe proprietà

per esempio `equals` deve implementare una relazione di equivalenza tra riferimenti ad oggetti che non sono `null`:
- è riflessiva
- è simmetrica
- è transitiva
- è consistente per ogni coppia di riferimenti non-`null` x e y , più invocazioni di `x.equals(y)` dovranno restituire consistentemente il valore `true` o `false` se durante le operazioni di confronto lo stato dei due oggetti non viene modificato
- per ogni riferimento non `null` x, `x.equals(null)` deve restituire sempre false

lo `hashcode` è un intero derivato dallo stato di un oggetto.
Gli `hash` code dovrebbero essere ottenuti in modo tale che se x e y non sono uguali allora `x.hascode()` e `y.hascode()` dovrebbero restituire valori differenti con alta probabilità.
La classe `Objects` mette a disposizione dei metodi di utilità per il calcolo degli `hashcode`:
- ogni volta che il metodo è invocato durante l'esecuzione il risultato del metodo `hashCode` restituisce in modo consistente lo stesso valore
- se due oggetti sono uguali rispetto al metodo `equals`, allora il metodo `hashCode` invocato sui due oggetti dovrà produrre lo stesso risultato
[`javadoc per objects`](https://docs.oracle.com/javase/8/docs/api/java/util/Objects.html)

#### come specifichiamo i contratti?
purtroppo java non supporta di specificare esplicitamente i contratti
per questa ragione tali informazioni vengono aggiunte nel `javadoc` di una classe 

Sia le invarianti di classe che i contratti di un metodo possono essere descritti:
- attraverso frasi nel linguaggio naturale
- attraverso i predicati di primo ordine (una formula logica che ritorna un booleano)
### la programmazione difensiva
nella scrittura di un metodo introduco dei costrutti sintattici per verificare se chi sta chiamando il metodo stia rispettando o meno il contratto.
con il termine _programmazione difensiva_ si intende quello stile di Coding dove all'inizio di ogni metodo vengono inseriti dei controlli per verificare se le condizioni del contratto siano rispettate. I controlli devono essere fatti solo nei metodi pubblici per tale ragione è buona norma limitare il numero di metodi pubblici forniti da una classe.

Cosa dobbiamo fare, però, quando un metodo identifica una violazione del contratto?
- Una possibilità è quella di restituire un codice di errore.
	 Questo è l'approccio usato in molti linguaggi di programmazione come, ad esempio, il C.
- Questa soluzione è però insoddisfacente per diverse ragioni
	- il valore di ritorno di un metodo è già usato per scopi diversi;
	- il chiamante deve gestire l'errore ed, eventualmente, restituire un errore diverso al suo contesto;
	- non è possibile verificare staticamente se tutti gli errori sono stati gestiti adeguatamente!
### le eccezioni
Per questa ragione in Java, come in altri linguaggi di programmazione moderni, gli errori vengono gestiti per mezzo delle eccezioni.
Un metodo segnala un problema serio andando a sollevare una eccezione.
Uno dei metodi nella catena delle chiamate sarà deputato a gestire tale eccezione.

### verificare i contratti
Come facciamo a garantire che una determinata classe soddisfi tutti i suoi
contratti?

Come sappiamo non esiste un modo per garantire che ogni tipologia di contratto sia verificata.
Un contratto potrebbe essere del tipo: "il metodo termina"

esistono comunque due approcci:
- l'analisi statica del codice ed il model checking
	- Java Path Finder
	- Java Modelling Language
- l'uso dei test (dimostra la presenza di un bug non l'assenza)

### Javadoc

Javadoc è uno strumento distribuito con il JDK per generare, a partire dal codice, la documentazione relativa ad una libreria/applicazione in formato HTML.
La documentazione generata verrà poi utilizzata dai vari IDE per mostrare il funzionamento dei metodi.
Il formato di Javadoc si basa su commenti opportunamente arricchiti per fornire informazioni.
La documentazione della libreria standard Java è costruita mediante [Javadoc](https://docs.oracle.com/javase/8/docs/technotes/tools/windows/javadoc.html).
un commento Javadoc è un commento che inizia con un doppio asterisco:
```java
/** commento javadoc **/
```
Il commento Javadoc deve essere inserito immediatamente prima dell'elemento che si vuole commentare. (metodi, campi, classi, ecc...)
All'interno del blocco di commento è possibile inserire dei tag che aggiungono
semantica alle informazioni.
```java
/**
 * Le istanze di questa classe vengono utilizzate
 * per rappresentare le coordinate di una casella.
 * 
 * @author Michele Loreti
**/
public class Position {
```

il Javadoc va scritto in inglese e bisogna evitare lettere accentate

ci vengono messi a disposizioni vari tag:

| Tag         | Descrizione                                                                                                       |
| ----------- | ----------------------------------------------------------------------------------------------------------------- |
| @author     | Nome dello sviluppatore.                                                                                          |
| @deprecated | Indica che l'elemento potrà essere eliminato da unaversione successiva del software.                              |
| @link       | Crea un collegamento ipertestuale alla documentazionelocale o a risorse esterna (tipicamente internet).           |
| @param      | Definisce i parametri di un metodo. Richiesto per ogni parametro.                                                 |
| @return     | Indica i valori di ritorno di un metodo. Questo tag non va usato per metodi o costruttori che restituiscono void. |
| @see        | Indica un'associazione a un altro metodo o classe.                                                                |
| @throws     | Indica eccezioni lanciate da un metodo.                                                                           |
Alcune considerazioni:
- la documentazione deve essere aggiunta ad ogni elemento pubblico;
- è più conveniente scrivere la documentazione in inglese.
